\section{Introduction}
% =============================================================================

Cryptocurrency mining is the process by which new transactions are verified
and added to the blockchain ledger. The core computational challenge is
known as \textbf{Proof of Work (PoW)}: miners must find a value (called a
\textit{nonce}) such that the hash of the block data produces a result that
satisfies a certain difficulty condition --- typically, the hash must be
numerically less than a \textit{target} value.

In Bitcoin, this involves computing the SHA-256 hash function twice
(\textit{double SHA-256}) over an 80-byte block header while iterating over
a 32-bit nonce space. The difficulty target is adjusted so that, on average,
a valid nonce is found every 10 minutes across the entire network.

\textbf{Pico Miner} is a proof-of-concept project that implements
\textbf{real Bitcoin double-SHA-256 mining} with midstate optimization,
accelerated on a Zynq-7020 FPGA using Vivado HLS 2019.1. This is not a
simulation or toy hash --- the SHA-256 implementation follows FIPS~180-4
exactly, the block headers are byte-for-byte copies from the real
blockchain, and the nonces found by the FPGA are the same nonces that were
originally mined in 2009.

The design is verified against \textbf{three real Bitcoin blocks}:
\begin{itemize}
    \item \textbf{Block~1} --- the first block after the genesis block
          (Satoshi Nakamoto, January~9, 2009)
    \item \textbf{Block~170} --- the first block containing a non-coinbase
          transaction (Satoshi sent 10~BTC to Hal Finney, January~12, 2009)
    \item \textbf{Block~181} --- an early block with two transactions
          (January~12, 2009)
\end{itemize}

\section{Background: How Bitcoin Mining Works}
% =============================================================================

\subsection{The Blockchain}

A blockchain is a distributed, append-only ledger where each block contains:
\begin{itemize}
	\item A reference (hash) to the previous block
	\item A set of transactions (represented by the Merkle root)
	\item A timestamp and difficulty bits
	\item A \textit{nonce} value
\end{itemize}

\subsection{The 80-Byte Block Header}

Every Bitcoin block header consists of exactly 80~bytes (640~bits):

\begin{table}[h]
	\centering
	\caption{Bitcoin Block Header Structure}
	\begin{tabular}{@{}lcl@{}}
		\toprule
		\textbf{Field}    & \textbf{Size (bytes)} & \textbf{Description} \\
		\midrule
		Version           & 4  & Block version number \\
		Previous hash     & 32 & SHA-256d hash of the previous block header \\
		Merkle root       & 32 & Hash of all transactions in the block \\
		Timestamp         & 4  & Unix timestamp \\
		Bits              & 4  & Compact representation of the target \\
		Nonce             & 4  & Value iterated by miners \\
		\bottomrule
	\end{tabular}
\end{table}

\subsection{Proof of Work}

The PoW mechanism requires that:
\begin{equation}
	\text{SHA-256}\bigl(\text{SHA-256}(\text{block\_header})\bigr) < \text{target}
\end{equation}

Since SHA-256 behaves as a pseudo-random oracle, the only way to find a
valid nonce is by \textbf{brute-force iteration} --- trying nonce values
one by one until the condition is met.

\subsection{Why FPGAs?}

FPGAs offer several advantages for PoW mining:
\begin{itemize}
	\item \textbf{Parallelism}: Multiple hash computations can run
	      simultaneously in hardware
	\item \textbf{Pipelining}: The compression rounds can be deeply
	      pipelined to achieve high throughput
	\item \textbf{Energy efficiency}: Dedicated hardware is far more
	      energy-efficient than general-purpose CPUs or GPUs
	\item \textbf{Low latency}: Direct hardware implementation avoids
	      instruction fetch/decode overhead
\end{itemize}

\section{SHA-256 Algorithm}
% =============================================================================

\subsection{Overview}

SHA-256 (Secure Hash Algorithm, 256-bit) is defined in FIPS~180-4.
It processes input in 512-bit (64-byte) blocks, maintaining an
eight-word (256-bit) state that is updated by a compression function
consisting of 64~rounds.

\subsection{Message Padding}

The input message is padded to a multiple of 512~bits:
\begin{enumerate}
	\item Append a single \texttt{1} bit
	\item Append \texttt{0} bits until the length is $\equiv 448 \pmod{512}$
	\item Append the original message length as a 64-bit big-endian integer
\end{enumerate}

For the 80-byte Bitcoin header, the padded message is 1024~bits (two 512-bit blocks).

\subsection{Compression Function}

Each 512-bit block is processed as follows:

\begin{algorithm}[H]
	\caption{SHA-256 Compression (one 64-byte block)}
	\begin{algorithmic}[1]
		\STATE \textbf{Input:} State $H[0..7]$, Message block $M[0..15]$
		\STATE \textbf{Message schedule expansion:}
		\FOR{$i = 0$ \TO $15$}
			\STATE $W[i] \leftarrow M[i]$
		\ENDFOR
		\FOR{$i = 16$ \TO $63$}
			\STATE $W[i] \leftarrow \sigma_1(W[i{-}2]) + W[i{-}7] + \sigma_0(W[i{-}15]) + W[i{-}16]$
		\ENDFOR
		\STATE \textbf{Initialise working variables:}
		\STATE $a,b,c,d,e,f,g,h \leftarrow H[0],H[1],\ldots,H[7]$
		\STATE \textbf{64 compression rounds:}
		\FOR{$i = 0$ \TO $63$}
			\STATE $T_1 \leftarrow h + \Sigma_1(e) + \text{Ch}(e,f,g) + K[i] + W[i]$
			\STATE $T_2 \leftarrow \Sigma_0(a) + \text{Maj}(a,b,c)$
			\STATE $h \leftarrow g;\; g \leftarrow f;\; f \leftarrow e;\; e \leftarrow d + T_1$
			\STATE $d \leftarrow c;\; c \leftarrow b;\; b \leftarrow a;\; a \leftarrow T_1 + T_2$
		\ENDFOR
		\STATE \textbf{Update state:}
		\STATE $H[i] \leftarrow H[i] + \{a,b,c,d,e,f,g,h\}$ for $i = 0..7$
	\end{algorithmic}
\end{algorithm}

Where the SHA-256 functions are:
\begin{align}
	\text{Ch}(e,f,g)  &= (e \wedge f) \oplus (\neg e \wedge g) \\
	\text{Maj}(a,b,c) &= (a \wedge b) \oplus (a \wedge c) \oplus (b \wedge c) \\
	\Sigma_0(a)        &= \text{ROTR}^2(a) \oplus \text{ROTR}^{13}(a) \oplus \text{ROTR}^{22}(a) \\
	\Sigma_1(e)        &= \text{ROTR}^6(e) \oplus \text{ROTR}^{11}(e) \oplus \text{ROTR}^{25}(e) \\
	\sigma_0(x)        &= \text{ROTR}^7(x) \oplus \text{ROTR}^{18}(x) \oplus \text{SHR}^3(x) \\
	\sigma_1(x)        &= \text{ROTR}^{17}(x) \oplus \text{ROTR}^{19}(x) \oplus \text{SHR}^{10}(x)
\end{align}

\subsection{Double SHA-256 for Bitcoin}

Bitcoin mining computes:
\[
	\text{hash} = \text{SHA-256}\bigl(\text{SHA-256}(\text{header}_{80\text{B}})\bigr)
\]

The first SHA-256 processes the 80-byte header (padded to 128~bytes = 2~blocks).
The second SHA-256 processes the resulting 32-byte digest (padded to 64~bytes = 1~block).
Total: \textbf{3 compression rounds} per nonce without optimization.

% =============================================================================
\section{Midstate Optimization}
% =============================================================================

\subsection{Key Insight}

The 80-byte block header spans two SHA-256 input blocks:
\begin{itemize}
	\item \textbf{Chunk~1} (bytes 0--63): version + previous hash + merkle root bytes 0--27
	\item \textbf{Chunk~2} (bytes 64--79 + padding): merkle root bytes 28--31 + timestamp + bits + \textbf{nonce}
\end{itemize}

Since the nonce is in chunk~2, the SHA-256 state after processing chunk~1
is \textbf{constant} across all nonce attempts. This intermediate state is
called the \textit{midstate}.

\subsection{ARM/FPGA Partitioning}

\begin{center}
\begin{tikzpicture}[

        text centered, minimum height=1.4cm, rounded corners},
    hwblock/.style={rectangle, draw, fill=orange!15, text width=6cm,

    \node[block, below=0.5cm of arm] (midstate) {

    };

    % FPGA side
    \node[hwblock, right=2cm of arm] (fpga) {FPGA Miner IP (PL)};
    \node[hwblock, below=0.5cm of fpga] (mining) {
        Per-nonce mining loop \\
        \footnotesize 2$\times$ SHA-256 compress (128 rounds)
    };

    % Connections
    \draw[arrow] (arm) -- node[above]{\footnotesize AXI-Lite} (fpga);
    \draw[arrow] (midstate) -- node[left]{\footnotesize } (arm);
    \draw[arrow] (fpga) -- node[right]{\footnotesize } (mining);

    % Labels
    \node[below=0.3cm of midstate, text width=6cm, align=center, font=\footnotesize] {
        Sends: midstate[8], chunk2\_tail[3], \\
        nonce\_start, nonce\_end, target\_hi

    };
\end{tikzpicture}
\end{center}

The ARM computes the midstate \textbf{once}, then the FPGA performs only:
\begin{enumerate}
	\item 64 compression rounds for chunk~2 (starting from midstate)
	\item 64 compression rounds for the second SHA-256
\end{enumerate}

This reduces the per-nonce work from 3~compressions to \textbf{2~compressions}
(128~rounds total), a 33\% reduction.

\section{System Architecture}
% =============================================================================

\subsection{HLS IP Interface}

The HLS IP function signature is:

\begin{lstlisting}[style=cstyle]
void pico_miner(
    unsigned int midstate[8],      // Precomputed SHA-256 state after chunk 1
    unsigned int chunk2_tail[3],   // merkle_tail, timestamp, bits
    unsigned int nonce_start,      // Beginning of nonce range
    unsigned int nonce_end,        // End of nonce range (exclusive)
    unsigned int target_hi,        // MSB of difficulty target
    unsigned int *found_nonce,     // Output: winning nonce (if found)
    unsigned int *status           // Output: 1 = found, 0 = not found
);
\end{lstlisting}

All ports are mapped to AXI-Lite (\texttt{s\_axilite}) bundled into a
single bus called \texttt{myaxi}. Vivado HLS 2019.1 maps array arguments
as individual scalar registers:
\begin{itemize}
	\item \texttt{midstate[8]} $\rightarrow$ \texttt{Set\_midstate\_0} \ldots \texttt{Set\_midstate\_7}
	\item \texttt{chunk2\_tail[3]} $\rightarrow$ \texttt{Set\_chunk2\_tail\_0} \ldots \texttt{Set\_chunk2\_tail\_2}
\end{itemize}

\subsection{Mining Operation Flow}

\begin{enumerate}
	\item The ARM parses the 80-byte block header and byte-swaps to big-endian
	\item The ARM computes the midstate: \texttt{sha256\_compress(H\_init, chunk1)}
	\item The ARM writes midstate[8], chunk2\_tail[3], nonce range, and target
	      to the FPGA via AXI-Lite
	\item The ARM issues \texttt{ap\_start}
	\item The FPGA iterates nonces:
	      \begin{enumerate}
		      \item Builds chunk~2 message block (tail + nonce + SHA-256 padding)
		      \item Compresses chunk~2 with the midstate $\rightarrow$ first hash
		      \item Hashes the first hash with SHA-256 initial values $\rightarrow$ final hash
		      \item If \texttt{final\_hash[0] <= target\_hi}: stores nonce, sets status
	      \end{enumerate}
	\item The ARM polls \texttt{ap\_done}, then reads \texttt{found\_nonce} and \texttt{status}
\end{enumerate}

\section{HLS Optimization Strategy}
% =============================================================================

\subsection{SHA-256 Compress Function}

The \texttt{sha256\_compress()} function is the critical inner kernel:

\begin{lstlisting}[style=cstyle]
static void sha256_compress(const unsigned int state_in[8],
                            const unsigned int W_in[16],
                            unsigned int state_out[8])
{
#pragma HLS INLINE off
    unsigned int W[64];
#pragma HLS ARRAY_PARTITION variable=W complete dim=1

    // Load + expand W (fully unrolled)
    // 64 compression rounds (pipelined II=1)
    compress:
    for (i = 0; i < 64; i++) {
#pragma HLS PIPELINE II=1
        // ... round computation
    }
}
\end{lstlisting}

Key optimizations:
\begin{itemize}
	\item \texttt{ARRAY\_PARTITION complete} on \texttt{W[64]}: all 64~words
	      accessible in parallel (no memory port contention)
	\item \texttt{UNROLL} on W load and expansion: computed combinationally
	\item \texttt{PIPELINE II=1} on the 64-round loop: one round per clock cycle
	\item \texttt{INLINE off}: keeps the compress function as a reusable module
	      (called twice per nonce -- once for chunk~2, once for the second hash)
\end{itemize}

\subsection{Mining Loop}

The top-level mining loop has no explicit pipeline pragma.
Each iteration calls \texttt{sha256\_compress} twice sequentially,
so each nonce takes approximately $2 \times 64 = 128$ clock cycles.

\subsection{Nonce Byte Order}

Bitcoin serialises nonces in \textbf{little-endian} (LE) in the block header.
SHA-256 processes words in \textbf{big-endian} (BE). The FPGA iterates the
BE nonce word directly. For example, Block~170's nonce:
\begin{itemize}
    \item LE (as serialised): \texttt{0x283e9e70}
    \item BE (SHA-256 word): \texttt{0x709e3e28}
\end{itemize}

The FPGA searches in a different order than a LE miner but covers the same
$2^{32}$ nonce space and produces identical hashes for any given nonce value.

\section{Verification Strategy}
% =============================================================================

The project employs a multi-level verification strategy to prove that the
FPGA performs real Bitcoin mining.

\subsection{HLS Testbench (6 Test Cases)}

The testbench (\texttt{pico\_miner\_tb.cpp}) contains six test cases:

\begin{enumerate}
	\item \textbf{NIST known vector}: SHA-256(``abc'') validated against the
	      official FIPS~180-4 test vector. Proves the SHA-256 implementation
	      is correct.

	\item \textbf{Block~1 full-range SW mining}: Mines Block~1 from nonce=0
	      in pure software ($\sim$31.8M iterations). This is the strongest
	      proof that the algorithm works: it finds a real Bitcoin nonce
	      starting from scratch, exactly as a real miner would.

	\item \textbf{Block~1 HW mining}: Calls the HLS accelerator with a
	      $\pm16$ nonce window around the known answer. Verifies the FPGA
	      hardware finds the correct nonce.

	\item \textbf{Block~170 HW mining}: Same $\pm16$ window test for
	      Block~170 (Satoshi $\rightarrow$ Hal Finney).

	\item \textbf{Block~181 HW mining}: Same $\pm16$ window test for
	      Block~181.

	\item \textbf{No-solution case}: Verifies the miner correctly reports
	      ``not found'' when no valid nonce exists in the search range.
\end{enumerate}

The narrow $\pm16$ window for HW tests keeps C/RTL co-simulation fast
(32~nonces $\times$ 128~cycles/nonce $\approx$ 4096~cycles per block).
The full-range search in Test~2 runs in pure software (no HW call) to
avoid cosim timeout.

\subsection{C Simulation (csim)}

All six tests run during \texttt{csim\_design}. The full-range Block~1
search (Test~2) takes $\sim$10~seconds in C~simulation.

\subsection{C/RTL Co-Simulation (cosim)}

After C synthesis, the generated RTL is verified against the C testbench.
The HW calls use narrow nonce windows ($\pm16$) to keep cosim runtime
manageable (the no-solution test searches 100~nonces).

\subsection{ARM Integration Test}

The ARM driver (\texttt{pico\_miner\_arm.c}) mines all three blocks on
real FPGA hardware with a $\pm4096$ nonce search window ($\sim$8192~nonces,
$\sim$10\,ms on the FPGA at 100\,MHz). For each block it:
\begin{enumerate}
	\item Parses the real 80-byte block header and byte-swaps to big-endian
	\item Computes the midstate on ARM
	\item Starts the HW miner and polls for completion
	\item Runs the SW golden model over the same range
	\item Compares HW vs.\ SW results bit-for-bit
	\item Verifies the found nonce matches the known blockchain nonce
\end{enumerate}

\section{Conclusion}
% =============================================================================

Pico Miner demonstrates that \textbf{real Bitcoin mining} can be implemented
on a Zynq-7020 FPGA using Vivado HLS. The design implements the complete
double-SHA-256 algorithm with midstate optimization, verified against three
actual blocks from the Bitcoin blockchain (Blocks~1, 170, and~181).

The project proves its correctness through multiple levels of evidence:
\begin{itemize}
    \item NIST FIPS~180-4 test vector validation (SHA-256 correctness)
    \item Full-range mining of Block~1 from nonce=0 ($\sim$31.8M iterations),
          proving the algorithm can find real Bitcoin nonces from scratch
    \item Hardware mining of three real blocks, with bit-exact SW/HW comparison
    \item Block hashes verified against the Bitcoin blockchain
\end{itemize}

The project showcases:
\begin{itemize}
	\item High-Level Synthesis of SHA-256 from C to RTL
	\item Midstate optimization for efficient ARM/FPGA partitioning
	\item AXI-Lite integration for PS/PL communication
	\item HLS directive optimization (pipeline II=1, array partition, unroll)
	\item Multi-level verification: NIST test vectors, real block mining,
	      SW/HW golden model comparison across 3 blocks
\end{itemize}

The estimated throughput of $\sim$780\,KH/s at 100\,MHz, while far below
modern Bitcoin ASICs, demonstrates the fundamental principles and
architecture of hardware-accelerated cryptocurrency mining using the
identical algorithm that secures the Bitcoin network.
