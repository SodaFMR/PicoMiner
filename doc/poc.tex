\bsection{Introducci\'on}{Introduction}
% =============================================================================

\begin{bilingual}
\espara{%
La miner\'ia de criptomonedas es el proceso mediante el cual se verifican las
transacciones y se a\~naden al registro distribuido conocido como \textit{blockchain}.
El reto computacional central se denomina \textbf{Prueba de Trabajo} (\textit{Proof of Work},
PoW): los mineros deben encontrar un valor llamado \textit{nonce} tal que el hash
del bloque de datos cumpla una condici\'on de dificultad --- t\'ipicamente, el hash
resultante debe ser num\'ericamente menor que un valor \textit{objetivo} (\textit{target}).}

\enpara{%
Cryptocurrency mining is the process by which transactions are verified
and added to the distributed ledger known as the blockchain.
The core computational challenge is called \textbf{Proof of Work} (PoW):
miners must find a value called a nonce such that the hash of the block
data satisfies a difficulty condition --- typically, the resulting hash
must be numerically less than a target value.}
\end{bilingual}

\begin{bilingual}
\espara{%
En Bitcoin, esto implica calcular la funci\'on hash SHA-256 dos veces
(\textit{double SHA-256}) sobre una cabecera de bloque de 80 bytes, iterando
sobre un espacio de nonces de 32 bits. El objetivo de dificultad se ajusta
para que, en promedio, se encuentre un nonce v\'alido cada 10 minutos en toda
la red.}

\enpara{%
In Bitcoin, this involves computing the SHA-256 hash function twice
(double SHA-256) over an 80-byte block header while iterating over a
32-bit nonce space. The difficulty target is adjusted so that, on average,
a valid nonce is found every 10 minutes across the entire network.}
\end{bilingual}

\begin{bilingual}
\espara{%
\textbf{Pico Miner} implementa la miner\'ia Bitcoin double-SHA-256 con optimizaci\'on
de midstate, acelerada en una FPGA Zynq-7020 mediante Vivado HLS 2019.1.
La demostraci\'on mina el Bloque 939260 de Bitcoin (4 de marzo de 2026,
dificultad 144.398.401.518.101) desde nonce=0 en aproximadamente 3 minutos,
con retroalimentaci\'on en tiempo real por terminal UART y LEDs del ZedBoard.}

\enpara{%
Pico Miner implements Bitcoin double-SHA-256 mining with midstate
optimization, accelerated on a Zynq-7020 FPGA using Vivado HLS 2019.1.
The demo mines Bitcoin Block 939260 (March 4, 2026,
difficulty 144,398,401,518,101) from nonce=0 in approximately 3 minutes,
with real-time feedback via UART terminal and ZedBoard LEDs.}
\end{bilingual}

\bsection{Fundamentos: Miner\'ia Bitcoin}{Background: Bitcoin Mining}
% =============================================================================

\bsubsection{La Blockchain}{The Blockchain}

\begin{bilingual}
\espara{%
Una blockchain es un registro distribuido y de solo escritura, donde cada bloque
contiene:
\begin{itemize}[leftmargin=2em]
    \item Una referencia (hash) al bloque anterior
    \item Un conjunto de transacciones (representadas por la ra\'iz de Merkle)
    \item Una marca de tiempo (\textit{timestamp}) y bits de dificultad
    \item Un valor \textit{nonce}
\end{itemize}}

\enpara{%
A blockchain is a distributed, append-only ledger where each block contains:
a reference (hash) to the previous block, a set of transactions (represented
by the Merkle root), a timestamp and difficulty bits, and a nonce value.}
\end{bilingual}

\bsubsection{Prueba de Trabajo}{Proof of Work}

\begin{bilingual}
\espara{%
El mecanismo de PoW requiere que:
\begin{equation}
    \text{SHA-256}\bigl(\text{SHA-256}(\text{cabecera\_bloque})\bigr) < \text{objetivo}
\end{equation}
Dado que SHA-256 se comporta como un or\'aculo pseudo-aleatorio, la \'unica forma
de encontrar un nonce v\'alido es mediante \textbf{iteraci\'on por fuerza bruta}:
probando valores de nonce uno por uno hasta cumplir la condici\'on.}

\enpara{%
The PoW mechanism requires that
$\text{SHA-256}(\text{SHA-256}(\text{block\_header})) < \text{target}$.
Since SHA-256 behaves as a pseudo-random oracle, the only way to find a
valid nonce is by brute-force iteration --- trying nonce values one by one
until the condition is met.}
\end{bilingual}

\bsubsection{Ventajas de las FPGAs}{Why FPGAs?}

\begin{bilingual}
\espara{%
Las FPGAs ofrecen varias ventajas para la miner\'ia PoW:
\begin{itemize}[leftmargin=2em]
    \item \textbf{Paralelismo}: M\'ultiples c\'alculos de hash simult\'aneos en hardware
    \item \textbf{Pipeline}: Las rondas de compresi\'on se pueden segmentar profundamente
    \item \textbf{Eficiencia energ\'etica}: Hardware dedicado es m\'as eficiente que CPUs o GPUs
    \item \textbf{Baja latencia}: Implementaci\'on directa sin overhead de b\'usqueda/decodificaci\'on de instrucciones
\end{itemize}}

\enpara{%
FPGAs offer parallelism (multiple simultaneous hash computations), deep
pipelining of compression rounds, energy efficiency compared to CPUs/GPUs,
and low latency through direct hardware implementation without instruction
fetch/decode overhead.}
\end{bilingual}

\bsection{Algoritmo SHA-256}{SHA-256 Algorithm}
% =============================================================================

\bsubsection{Visi\'on General}{Overview}

\begin{bilingual}
\espara{%
SHA-256 (Secure Hash Algorithm, 256 bits) est\'a definido en FIPS 180-4.
Procesa la entrada en bloques de 512 bits (64 bytes), manteniendo un estado
de ocho palabras de 32 bits (256 bits en total) que se actualiza mediante
una funci\'on de compresi\'on de 64 rondas.}

\enpara{%
SHA-256 (Secure Hash Algorithm, 256-bit) is defined in FIPS 180-4.
It processes input in 512-bit (64-byte) blocks, maintaining an eight-word
(256-bit) state that is updated by a compression function consisting of
64 rounds.}
\end{bilingual}

\bsubsection{Relleno del Mensaje}{Message Padding}

\begin{bilingual}
\espara{%
El mensaje de entrada se rellena hasta ser m\'ultiplo de 512 bits:
\begin{enumerate}[leftmargin=2em]
    \item A\~nadir un bit \texttt{1}
    \item A\~nadir bits \texttt{0} hasta que la longitud sea $\equiv 448 \pmod{512}$
    \item A\~nadir la longitud original del mensaje como un entero de 64 bits en big-endian
\end{enumerate}
Para la cabecera Bitcoin de 80 bytes, el mensaje rellenado tiene 1024 bits (dos bloques de 512 bits).}

\enpara{%
The input message is padded to a multiple of 512 bits by appending a
\texttt{1} bit, then \texttt{0} bits until the length is $\equiv 448
\pmod{512}$, and finally the original message length as a 64-bit
big-endian integer. For the 80-byte Bitcoin header, the padded message
is 1024 bits (two 512-bit blocks).}
\end{bilingual}

\bsubsection{Funci\'on de Compresi\'on}{Compression Function}

\begin{bilingual}
\espara{%
Cada bloque de 512 bits se procesa seg\'un el Algoritmo~\ref{alg:sha256compress}.
Las funciones SHA-256 utilizadas son:}

\enpara{%
Each 512-bit block is processed according to Algorithm~\ref{alg:sha256compress}.
The SHA-256 functions used are:}
\end{bilingual}

\begin{algorithm}[H]
    \caption{SHA-256 Compress / Compresi\'on SHA-256}
    \label{alg:sha256compress}
    \begin{algorithmic}[1]
        \STATE \textbf{Input / Entrada:} Estado $H[0..7]$, Bloque $M[0..15]$
        \STATE \textbf{Expansi\'on del calendario de mensajes / Message schedule:}
        \FOR{$i = 0$ \TO $15$}
            \STATE $W[i] \leftarrow M[i]$
        \ENDFOR
        \FOR{$i = 16$ \TO $63$}
            \STATE $W[i] \leftarrow \sigma_1(W[i{-}2]) + W[i{-}7] + \sigma_0(W[i{-}15]) + W[i{-}16]$
        \ENDFOR
        \STATE \textbf{Inicializar variables / Initialise working variables:}
        \STATE $a,b,c,d,e,f,g,h \leftarrow H[0],H[1],\ldots,H[7]$
        \STATE \textbf{64 rondas de compresi\'on / 64 compression rounds:}
        \FOR{$i = 0$ \TO $63$}
            \STATE $T_1 \leftarrow h + \Sigma_1(e) + \text{Ch}(e,f,g) + K[i] + W[i]$
            \STATE $T_2 \leftarrow \Sigma_0(a) + \text{Maj}(a,b,c)$
            \STATE $h \leftarrow g;\; g \leftarrow f;\; f \leftarrow e;\; e \leftarrow d + T_1$
            \STATE $d \leftarrow c;\; c \leftarrow b;\; b \leftarrow a;\; a \leftarrow T_1 + T_2$
        \ENDFOR
        \STATE \textbf{Actualizar estado / Update state:}
        \STATE $H[i] \leftarrow H[i] + \{a,b,c,d,e,f,g,h\}$ for $i = 0..7$
    \end{algorithmic}
\end{algorithm}

\begin{align}
    \text{Ch}(e,f,g)  &= (e \wedge f) \oplus (\neg e \wedge g) \\
    \text{Maj}(a,b,c) &= (a \wedge b) \oplus (a \wedge c) \oplus (b \wedge c) \\
    \Sigma_0(a)        &= \text{ROTR}^2(a) \oplus \text{ROTR}^{13}(a) \oplus \text{ROTR}^{22}(a) \\
    \Sigma_1(e)        &= \text{ROTR}^6(e) \oplus \text{ROTR}^{11}(e) \oplus \text{ROTR}^{25}(e) \\
    \sigma_0(x)        &= \text{ROTR}^7(x) \oplus \text{ROTR}^{18}(x) \oplus \text{SHR}^3(x) \\
    \sigma_1(x)        &= \text{ROTR}^{17}(x) \oplus \text{ROTR}^{19}(x) \oplus \text{SHR}^{10}(x)
\end{align}

\bsubsection{Doble SHA-256 para Bitcoin}{Double SHA-256 for Bitcoin}

\begin{bilingual}
\espara{%
La miner\'ia Bitcoin calcula:
\[
    \text{hash} = \text{SHA-256}\bigl(\text{SHA-256}(\text{cabecera}_{80\text{B}})\bigr)
\]
El primer SHA-256 procesa la cabecera de 80 bytes (rellenada a 128 bytes = 2 bloques).
El segundo SHA-256 procesa el resultado de 32 bytes (rellenado a 64 bytes = 1 bloque).
Total sin optimizaci\'on: \textbf{3 compresiones} por nonce.}

\enpara{%
Bitcoin mining computes
$\text{hash} = \text{SHA-256}(\text{SHA-256}(\text{header}_{80\text{B}}))$.
The first SHA-256 processes the 80-byte header (padded to 128 bytes = 2 blocks).
The second SHA-256 processes the resulting 32-byte digest (padded to 64 bytes = 1 block).
Total without optimization: 3 compressions per nonce.}
\end{bilingual}

\bsection{Optimizaci\'on de Midstate}{Midstate Optimization}
% =============================================================================

\bsubsection{Idea Clave}{Key Insight}

\begin{bilingual}
\espara{%
La cabecera de 80 bytes abarca dos bloques de entrada SHA-256:
\begin{itemize}[leftmargin=2em]
    \item \textbf{Chunk 1} (bytes 0--63): versi\'on + hash previo + ra\'iz de Merkle bytes 0--27
    \item \textbf{Chunk 2} (bytes 64--79 + relleno): ra\'iz de Merkle bytes 28--31 + timestamp + bits + \textbf{nonce}
\end{itemize}
Dado que el nonce est\'a en el chunk 2, el estado SHA-256 tras procesar el chunk 1
es \textbf{constante} para todos los intentos de nonce. Este estado intermedio se
denomina \textit{midstate}.}

\enpara{%
The 80-byte header spans two SHA-256 input blocks. Chunk 1 (bytes 0--63) contains
version + previous hash + merkle root bytes 0--27. Chunk 2 (bytes 64--79 + padding)
contains merkle root bytes 28--31 + timestamp + bits + nonce. Since the nonce is in
chunk 2, the SHA-256 state after processing chunk 1 is constant across all nonce
attempts. This intermediate state is called the midstate.}
\end{bilingual}

\begin{table}[h]
    \centering
    \caption{\es{Divisi\'on de la cabecera para midstate} /
             \en{Header partitioning for midstate}}
    \rowcolors{2}{paleblue}{white}
    \begin{tabular}{@{}lccl@{}}
        \toprule
        \textbf{Chunk} & \textbf{Bytes} & \textbf{\es{Contenido}} & \textbf{\es{Procesado por} / \en{Processed by}} \\
        \midrule
        Chunk 1 & 0--63 & version + prev\_hash + merkle[0..27] & ARM (una vez / once) \\
        Chunk 2 & 64--79 + padding & merkle[28..31] + ts + bits + \textbf{nonce} & FPGA (por nonce / per nonce) \\
        \bottomrule
    \end{tabular}
\end{table}

\bsubsection{Partici\'on ARM/FPGA}{ARM/FPGA Partitioning}

\begin{bilingual}
\espara{%
El ARM calcula el midstate \textbf{una sola vez}, y luego la FPGA realiza \'unicamente:
\begin{enumerate}[leftmargin=2em]
    \item 64 rondas de compresi\'on para el chunk 2 (partiendo del midstate)
    \item 64 rondas de compresi\'on para el segundo SHA-256
\end{enumerate}
Esto reduce el trabajo por nonce de 3 compresiones a \textbf{2 compresiones}
(128 rondas en total), una reducci\'on del 33\%.}

\enpara{%
The ARM computes the midstate once, then the FPGA performs only 64 compression
rounds for chunk 2 (starting from the midstate) and 64 compression rounds for
the second SHA-256. This reduces the per-nonce work from 3 compressions to
2 compressions (128 rounds total), a 33\% reduction.}
\end{bilingual}

\bsubsection{Flujo de Operaci\'on de Miner\'ia}{Mining Operation Flow}

\begin{bilingual}
\espara{%
\begin{enumerate}[leftmargin=2em]
    \item El ARM analiza la cabecera de 80 bytes y la convierte a big-endian
    \item El ARM calcula el midstate: \texttt{sha256\_compress(H\_init, chunk1)}
    \item El ARM escribe midstate[8], chunk2\_tail[3], rango de nonces y target en la FPGA v\'ia AXI-Lite
    \item El ARM env\'ia \texttt{ap\_start}
    \item La FPGA itera nonces:
    \begin{enumerate}
        \item Construye el bloque de mensaje del chunk 2 (tail + nonce + relleno SHA-256)
        \item Comprime chunk 2 con el midstate $\rightarrow$ primer hash
        \item Hashea el primer hash con valores iniciales SHA-256 $\rightarrow$ hash final
        \item Si \texttt{final\_hash[7] <= target\_hi}: almacena nonce, establece status
    \end{enumerate}
    \item El ARM sondea \texttt{ap\_done}, luego lee \texttt{found\_nonce} y \texttt{status}
\end{enumerate}}

\enpara{%
1. ARM parses the 80-byte header and byte-swaps to big-endian.
2. ARM computes midstate.
3. ARM writes parameters to FPGA via AXI-Lite.
4. ARM issues ap\_start.
5. FPGA iterates nonces, performing double-SHA-256 and checking difficulty.
6. ARM polls ap\_done, then reads found\_nonce and status.}
\end{bilingual}

\bsection{Estrategia de Optimizaci\'on HLS}{HLS Optimization Strategy}
% =============================================================================

\bsubsection{Funci\'on de Compresi\'on SHA-256}{SHA-256 Compress Function}

\begin{bilingual}
\espara{%
La funci\'on \texttt{sha256\_compress()} es el n\'ucleo computacional cr\'itico.
Las optimizaciones clave aplicadas son:}

\enpara{%
The sha256\_compress() function is the critical inner kernel.
Key optimizations applied are:}
\end{bilingual}

\begin{lstlisting}[style=cstyle]
static void sha256_compress(const unsigned int state_in[8],
                            const unsigned int W_in[16],
                            unsigned int state_out[8])
{
#pragma HLS INLINE off
    unsigned int W[64];
#pragma HLS ARRAY_PARTITION variable=W complete dim=1

    // Carga + expansion de W (completamente desenrollado)
    // 64 rondas de compresion (pipeline II=1)
    compress:
    for (i = 0; i < 64; i++) {
#pragma HLS PIPELINE II=1
        // ... calculo de ronda / round computation
    }
}
\end{lstlisting}

\begin{bilingual}
\espara{%
\begin{itemize}[leftmargin=2em]
    \item \texttt{ARRAY\_PARTITION complete} en \texttt{W[64]}: las 64 palabras accesibles en paralelo (sin contenci\'on de puertos de memoria)
    \item \texttt{UNROLL} en carga y expansi\'on de W: calculado combinacionalmente
    \item \texttt{PIPELINE II=1} en el bucle de 64 rondas: una ronda por ciclo de reloj
    \item \texttt{INLINE off}: mantiene la funci\'on compress como m\'odulo reutilizable (se llama dos veces por nonce)
\end{itemize}}

\enpara{%
ARRAY\_PARTITION complete on W[64] enables parallel access.
UNROLL on W load/expansion computes combinationally.
PIPELINE II=1 on the 64-round loop produces one round per clock cycle.
INLINE off keeps the compress function as a reusable module (called twice per nonce).}
\end{bilingual}

\bsubsection{Bucle de Miner\'ia}{Mining Loop}

\begin{bilingual}
\espara{%
El bucle de miner\'ia de nivel superior no tiene pragma de pipeline expl\'icito.
Cada iteraci\'on llama a \texttt{sha256\_compress} dos veces secuencialmente,
por lo que cada nonce tarda aproximadamente $2 \times 64 = 128$ ciclos de reloj.}

\enpara{%
The top-level mining loop has no explicit pipeline pragma. Each iteration
calls sha256\_compress twice sequentially, so each nonce takes approximately
$2 \times 64 = 128$ clock cycles.}
\end{bilingual}

\bsubsection{Orden de Bytes del Nonce}{Nonce Byte Order}

\begin{bilingual}
\espara{%
Bitcoin serializa los nonces en \textbf{little-endian} (LE) en la cabecera del bloque.

palabra de nonce en BE. La FPGA busca en un orden diferente a un minero LE, pero
cubre el mismo espacio de $2^{32}$ nonces y produce hashes id\'enticos para cualquier
valor de nonce dado.}

\enpara{%

processes words in big-endian (BE). The FPGA iterates the BE nonce word directly,
searching in a different order than a LE miner but covering the same $2^{32}$
nonce space and producing identical hashes for any given nonce value.}
\end{bilingual}

\enpara{%

Each nonce requires two compressions, resulting in approximately 128 cycles
per nonce, yielding approximately 781 KH/s throughput.}

\bsubsection{Comprobaci\'on de Dificultad}{Difficulty Check}
